# 編輯器視覺重新設計 — Design Spec

**Date:** 2026-07-05
**Status:** Approved (brainstorming complete, ready for implementation planning)

## 背景

古琴聲光秀編輯器（`src/App.tsx` 及其六個子元件：`TransportBar`、`StatusBar`、`PositionStrip`、`GuqinDisplay`、`Timeline`、`ExportPanel`）目前完全沒有樣式 — 所有元件都是預設瀏覽器樣式的 `<div>` 直向堆疊，唯一的視覺客製化是元件內少數幾處 inline `style={{...}}`（如 `PositionStrip`、`GuqinDisplay` 的定位）。專案沒有任何 `.css` 檔案。

本次重新設計的目標：在不改變現有功能與資料流的前提下，給編輯器一個統一、有意圖的視覺外觀與版面配置，並為即將加入的 ADSR（音量包絡）編輯器與走手音（滑音路徑）編輯器預留版面位置。

## 美術方向：水墨古卷

米宣紙底、墨黑文字、硃砂紅點綴、金黃/赭石作為強調色（弦線、選中標記）。細筆劃邊框取代預設瀏覽器邊框，襯線字體（中文字型優先 serif，如 Noto Serif TC / Songti TC）。整體感覺內斂、有文化質感，避免高飽和度的「App 感」配色。

### 色票

| 用途 | 色值 | 說明 |
|---|---|---|
| 背景 | `#f3ede1` | 米宣紙底 |
| 面板背景 | `#fbf7ee` | 略亮於背景，區塊感 |
| 邊框（實） | `#c9b98f` | 主邊框、標題底線 |
| 邊框（虛/淡） | `#ddd0ac` | 面板內部細分隔線 |
| 主文字 | `#2b2621` | 墨黑 |
| 次要文字 | `#6b5f45` | 標籤、次要說明 |
| 強調紅 | `#a63b2e` | 選中標記、底線、未完成音警示、走手音圖表線 |
| 強調金 | `#b8944a` | 按鈕邊框/文字、弦線、徽位刻度基準線 |

字體：`"Noto Serif TC", "Songti TC", serif`（若系統無襯線中文字型則退回瀏覽器預設 serif）。

### 元件視覺對照

- **狀態列 / StatusBar**：目前編輯欄位用底線標示的做法（`textDecoration: underline`）保留，底線顏色改用強調紅（呼應水墨古卷的硃砂點綴），取代預設黑色底線。
- **音清單 / Timeline**：每筆一列，虛線分隔（`border-bottom` 用邊框淡色），選中列加粗，未完成音的文字顏色改用強調紅（目前是 `#e08`，色相調整為統一色票的 `#a63b2e`）。
- **弦線工作區 / PositionStrip**：弦線本身用金色，徽位刻度用小圓點（次要文字色），選中音的標記用強調紅並帶淡淡的光暈（`box-shadow`）。
- **展示區 / GuqinDisplay**：無額外裝飾，`guqin-vector.svg` 本身即為視覺主體；標記點维持現有暖黃色（`#ffe066`）不變，因為那是疊加在琴圖上的「光點」示意，屬於展示內容本身而非介面裝飾。

## 版面配置

放棄目前的「六塊等權重直向堆疊」，改為三個垂直區段，由上而下依「常駐提示/操作 → 主要視覺內容 → 選中音細節編輯」排列：

```
┌─────────────────────────────────────────────────────────┐
│ 標題／播放列／狀態列／匯出匯入（同一條頂部區塊，文字與按鈕為主） │
├───────────────────────────────┬───────────────────────────┤
│                               │                           │
│  展示區（GuqinDisplay，         │   音清單（Timeline）        │
│  guqin-vector.svg 疊加標記）    │                           │
│  — 依圖片實際比例撐開高度，       │                           │
│    不強制固定高度、不留空白      │                           │
│                               │                           │
├───────────────────────────────┴───────────────────────────┤
│ 工作區 · 弦線位置（PositionStrip）— 全寬，不收窄              │
├─────────────────────────────┬───────────────────────────────┤
│ 包絡（ADSR）— 佔位面板         │ 走手音路徑 — 佔位面板（僅按音） │
└─────────────────────────────┴───────────────────────────────┘
```

### 各區段規則

1. **頂部區塊**：`h1` 標題、`TransportBar`（檔名/播放按鈕/時間）、`StatusBar`（選中音資訊）、`ExportPanel`（匯出/匯入）並排或緊鄰排列在同一視覺區塊內，字級較小、視為「提示與操作文字」而非獨立卡片。彼此之間用細分隔或留白區分，不需要各自的邊框面板。
2. **展示區＋音清單（雙欄）**：左欄是 `GuqinDisplay`（比重較大），右欄是 `Timeline`（音清單）。左欄高度由 `guqin-vector.svg` 的實際顯示尺寸決定，**不設定固定高度**，右欄內容較多時用自己的捲動或自然撐開，不得透過 flex 拉伸把左欄撐出多餘空白（即左右兩欄不使用 `align-items: stretch`，改用 `flex-start`）。
3. **工作區 · 弦線位置**：`PositionStrip`，全寬顯示，不再收窄成左欄的一部分。
4. **包絡／走手音（雙欄，全寬）**：兩個新面板，這次**只做版面佔位**（面板框架＋標籤文字，如「包絡（ADSR）」「走手音路徑（僅按音）」），不實作內部互動或資料模型。面板保留可辨識的位置與比例（各佔一半寬），讓未來實作時只需要把佔位內容換成真正的編輯器元件，不需要調整外層版面。走手音面板在概念上只對按音（`type === 'an'`）有意義，但這次不需要做條件顯示邏輯 — 面板本身固定顯示即可，是否依 note type 顯示/隱藏留給未來實作 ADSR/走手音編輯器時再處理。

### 響應式

不特別處理窄螢幕/手機版面（此編輯器predominantly 桌面使用情境），維持固定寬度佈局即可，與現有元件的固定尺寸假設（如 `PositionStrip` 的 `W=800`、`GuqinDisplay` 的 `480x160`）一致。

## 技術做法：單一共用 CSS 檔案

新增 `src/styles.css`，在 `App.tsx` 頂部 `import './styles.css'` 一次。所有元件的 class 命名沿用目前已存在的 kebab-case 慣例（`.status-bar`、`.transport`、`.timeline`、`.export`），新增的頂部區塊與雙欄/全寬容器另外取新 class（如 `.top-bar`、`.main-panels`、`.workspace-strip`、`.workspace-drawer`）。

不引入 CSS Modules、CSS-in-JS 或任何新的建構工具設定 — 專案規模（8 個元件）不需要，維持與目前「零額外相依套件」的簡潔性一致。各元件現有的 inline style（如 `PositionStrip`、`GuqinDisplay` 內以 JS 算出的動態座標）維持 inline，因為那些是資料驅動的動態定位，不適合搬進靜態 CSS；只有純粹的靜態外觀（顏色、邊框、字體、間距、版面排列）搬進 `styles.css`。

## 範圍界定

**這次要做的：**
- 新增 `src/styles.css`，套用上述色票、字體、版面規則到六個現有元件
- 調整 `App.tsx` 的 JSX 結構以符合新版面（頂部區塊 / 雙欄主區 / 全寬工作區 / 全寬佔位雙欄）
- 新增兩個佔位面板（包絡、走手音），僅有框架與標籤，無互動功能、無資料模型變更
- `Timeline` 未完成音的顏色從 `#e08` 改為統一色票的 `#a63b2e`

**這次不做的：**
- ADSR / 走手音編輯器的實際互動功能與資料模型（未來另立專案）
- 響應式/行動裝置版面
- 任何現有元件的功能行為變更（快捷鍵、store、audio engine 邏輯完全不動）
- 依 note type 顯示/隱藏走手音面板的條件邏輯

## 測試與驗證

視覺樣式變更沒有適合的自動化測試，改用手動驗證：`npm run dev` 開啟後確認六個既有元件功能不受影響（載入音檔播放、快捷鍵插入音、狀態列底線顯示、弦線工作區點擊設定位置、音清單選取/刪除、匯出匯入 round-trip），並確認新版面在展示區圖片載入後不出現異常空白或跑版。既有的 Vitest 測試套件（`model/`、`state/`、`hooks/`）不涉及樣式，預期全數維持通過。
