# 古琴聲光秀 — 第一步「軟體版」實作計畫

## Context / 背景

`計畫.md` 描述了一個三階段專案:用七條 LED 光帶代表古琴七弦,隨音樂展現散音、泛音、按音三種音色與指法。本計畫**聚焦第一步:互動式編輯/播放軟體**——讓使用者匯入一段現成的古琴錄音,一邊播放一邊逐音標記,每個音可設定音色、ADSR 包絡、徽位/按點與走手音軌跡,即時在介面上的古琴看到光帶特效,最後匯出「光帶參數檔」供第二步實體 LED 使用。

關鍵共識(已與使用者確認):
- **平台**:網頁應用(理由見下)。
- **音樂來源**:匯入現成音檔(mp3/wav),非即時合成。
- **光帶映射模型**:每條弦 = 一條 1D 光帶;**光點位置 = 弦上的彈奏點**,**亮度 = ADSR 包絡**,**走手音 = 光點沿弦移動**。
- **目標等級**:認真的展出/裝置 → 需穩定、精確、且資料結構預留實體擴展。
- **減字譜**:第一步純手動標記,但資料結構**預留減字譜欄位**,為第三步 AI(解決同音異位)鋪路。

---

## 1. 平台與技術選型

**推薦:網頁應用(React + TypeScript + Vite)。**

理由:
- **Web Audio API** 提供樣本級精確的時鐘(`AudioContext.currentTime`),這對「音樂時間戳」與「光效同步」是剛需,優於一般 `<audio>` 元素。
- **Canvas / WebGL** 能以 60fps 流暢渲染七弦 + 十三徽 + 光帶特效。
- 跨平台、零安裝;展出時可用 kiosk 模式全螢幕瀏覽器執行。編輯工具與第二步的實體播放本就該解耦——軟體只負責「產生參數檔」。
- 你略懂程式,Web 生態(TypeScript、現成函式庫)學習曲線最平緩、可參考資源最多。

建議技術棧:

| 用途 | 選擇 | 備註 |
|------|------|------|
| 框架 / 建置 | React + TypeScript + Vite | 元件化、型別安全 |
| 音訊播放與時鐘 | Web Audio API(`decodeAudioData` → `AudioBufferSourceNode`) | 精確同步基礎 |
| 波形顯示 | wavesurfer.js,或自繪解碼後的 `AudioBuffer` | 時間軸定位 |
| 古琴與光帶渲染 | Canvas 2D(起步),如需更炫光效再升級 PixiJS/WebGL | |
| 狀態管理 | Zustand | 輕量,易做 undo/redo |
| 曲線編輯器(ADSR / 走手音) | 自製 SVG 元件 + 可拖曳控制點 | 見 §4 |
| 存檔 / 匯出 | JSON 匯出入 + localStorage 自動存檔 | 見 §5 |

> 若日後想打包成桌面程式,React 程式可直接用 Electron/Tauri 包裝,不需重寫。

## 2. 核心資料模型

整個專案 = 一個音檔參照 + 一串「音事件(Note)」。建議型別(TypeScript):

```ts
interface Project {
  version: string;
  audio: { fileName: string; durationSec: number; sampleRate: number };
  notes: Note[];
  meta?: { title?: string; tuning?: number[] }; // 預留:七弦定音
}

interface Note {
  id: string;
  startTime: number;          // 音樂中的絕對時間(秒)
  string: 1|2|3|4|5|6|7;       // 哪一條弦
  type: 'san' | 'fan' | 'an'; // 散音 / 泛音 / 按音
  position: number;           // 弦上彈奏點,正規化 0..1(或以徽為單位 1..13);散音可為 null/整弦
  hui?: number;               // 泛音專用:1..13 徽位
  adsr: EnvelopePoint[];      // ADSR 包絡,可拖曳控制點(見 §4)
  slide?: SlidePoint[];       // 走手音軌跡(僅按音);空陣列=無走手音
  jianzipu?: string;          // 預留減字譜欄位(第三步用),第一步可手填或留空
  color?: string;             // 預留:該音光色
}

interface EnvelopePoint { t: number; level: number; } // t:相對音起點秒數, level:0..1 亮度
interface SlidePoint    { t: number; position: number; } // t:相對秒數, position:弦上位置 0..1
```

設計要點:
- `position` 採**正規化 0..1**(0=岳山/彈弦端,1=龍齦端),好處是與弦的實體長度脫鉤,第二步可直接乘以該弦 LED 數量得到像素索引。徽位則另存 `hui`(整數),UI 顯示時換算。
- `adsr` 與 `slide` 都是**控制點陣列**而非固定四參數,這樣曲線編輯器可自由插點;渲染時在點之間做插值(建議 Catmull-Rom 或三次貝茲)。
- `jianzipu` 現在是純字串欄位即可,不需在第一步解析減字譜;它的存在讓第三步 AI 能對齊「音 ↔ 譜字」。

## 3. 古琴領域與視覺對應規則

三種音色的視覺語法(此為**設計提案,實作前可再微調**):

- **散音(整弦空弦音)**:無左手按點。視覺上整條弦光帶一起亮起(或從岳山端快速掃過),亮度走 ADSR;音色最飽滿,可用最高基礎亮度/暖色。
- **泛音(在徽位節點輕觸)**:只在 `hui` 指定的徽位節點出現一個清亮、帶微閃爍(bell-like)的光點;古琴僅十三徽可發泛音,UI 應限制只能選徽位。
- **按音(壓弦實音)**:在 `position` 按點出現光點,亮度走 ADSR。若有 `slide`(走手音),光點沿弦移動;移動過程的「虛音」以**較低亮度/拖尾**渲染,與起始「實音」的高亮形成對比。走手音常見指法(綽、注、上、下、吟、猱)都能用 position-over-time 曲線表達。

七弦排列、十三徽位置需依古琴實際比例繪製(徽為泛音節點,七徽在弦中點)。定音預留於 `meta.tuning`。

## 4. UI 元件分解

各元件職責單一、介面清楚:

1. **播放列 `TransportBar`** — 載入音檔、播放/暫停、目前時間、播放速度(慢速便於逐音標記)、波形 + 播放游標。資料來源:Web Audio 時鐘。
2. **古琴畫布 `GuqinCanvas`** — 繪製七弦十三徽;**即時**讀取播放時鐘,根據 `notes` 渲染當下的光帶特效(亮起/移動/消失);同時可**點擊輸入**:點某條弦的某位置 → 在當前時間建立一個 Note。
3. **時間軸 / 音清單 `Timeline`** — 沿時間顯示所有已標記的音(每弦一軌),可點選、拖曳調整 `startTime`、刪除。
4. **音參數檢視器 `NoteInspector`** — 選中某音後編輯:音色型別、弦、位置/徽位、**ADSR 曲線編輯器**、**走手音軌跡編輯器**、減字譜欄位。
5. **匯出面板 `ExportPanel`** — 匯出/匯入 Project JSON;預留「編譯成 LED 影格」選項(第二步)。

**曲線編輯器(ADSR 與走手音共用一個底層元件)**:SVG 畫布 + 可用滑鼠新增/拖曳/刪除控制點,點間插值即時預覽。ADSR 的 Y 軸是亮度(0..1),走手音的 Y 軸是弦上位置(0..1),X 軸都是相對音起點的時間。

## 5. 匯出格式(銜接第二步)

第一步主要匯出 **Project JSON**(上方 `Project` 結構),它同時是存檔與第二步的輸入。

為第二步預留一個**可選的「編譯」輸出**:把以時間為單位的 notes,取樣成**以 LED 影格為單位**的逐燈亮度序列(例如每秒 30/60 影格,每條弦 N 顆燈)。這層轉換邏輯第一步可先不做,但 JSON 結構要讓它**容易實作**——這正是 `position` 正規化、`adsr`/`slide` 用時間軸控制點的原因。

## 6. 分階段建置順序(里程碑)

每個里程碑都應可獨立執行與驗證:

1. **骨架 + 音訊**:Vite/React/TS 專案;載入音檔、Web Audio 播放、波形 + 播放游標。
2. **靜態古琴畫布**:正確比例繪製七弦十三徽。
3. **最簡音事件**:點弦 → 在當前時間建立散音 Note;`Timeline` 顯示;可刪除。
4. **音色三型 + 位置**:散音/泛音/按音切換;泛音限徽位;按音可選按點。
5. **ADSR 曲線編輯器**:控制點拖曳 + 插值。
6. **即時光帶特效**:`GuqinCanvas` 隨播放時鐘渲染亮度/位置,音播完即消失。
7. **走手音軌跡編輯器 + 虛音渲染**:按音光點沿弦移動、虛音拖尾。
8. **減字譜欄位 + JSON 匯入/匯出 + 自動存檔**。
9. **打磨**:undo/redo、播放速度、時間量化(quantize)、鍵盤快捷鍵。

第 1–6 個里程碑完成即是一個「可用的原型」;7–9 讓它達到「認真展出」品質。

## 7. 第二步 / 第三步 方向性建議(本計畫不深入)

- **第二步(實體 LED)**:七弦各約 120cm,適合用**可定址 LED 燈條**(WS2812B/SK6812,常見 60 或 144 LED/m;144/m 在 120cm ≈ 173 顆,解析度高、走手音更平滑)。控制器可用 **ESP32**(內建 WiFi,適合做即時同步)搭配 FastLED/WLED 之類函式庫;七條燈條的資料量需評估是否分多個資料腳位或用電平轉換。第二步的核心工作是把第一步的 Project JSON「編譯」成逐影格逐燈亮度,並與音檔同步播放。**此步需另立計畫並先做硬體採購/打樣評估。**
- **第三步(AI 自動產生參數)**:因古琴同音異位,AI 需同時參考減字譜——這正是第一步預留 `jianzipu` 欄位的原因。先用第一步人工標記累積一批「音檔 + 參數 + 減字譜」資料當訓練/驗證集,再評估方法(音訊事件偵測 + 譜面對齊)。**此步亦需另立計畫。**

## 9. 驗證方式

- **里程碑 1**:能載入一段古琴 mp3,播放/暫停/拖曳,游標與聲音同步(肉眼/耳朵對齊誤差 < ~50ms)。
- **里程碑 3–4**:播放中點某弦某位置能建立正確型別與時間的 Note,`Timeline` 與 JSON 都反映正確;泛音被限制只能選徽位。
- **里程碑 6–7**:逐音播放時,光點在**正確的弦、正確的位置**亮起,亮度隨 ADSR 變化,按音走手音時光點平滑移動且虛音較暗;音結束後光點消失。
- **里程碑 8**:匯出 JSON → 重新匯入 → 畫面與資料完全一致(round-trip 無損)。
- 端到端:用一小段真實古琴錄音(含散/泛/按各至少一個音、含一次走手音)逐音標記完成,匯出檔案結構正確且可重現視覺。
