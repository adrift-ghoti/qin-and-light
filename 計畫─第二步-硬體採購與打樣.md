# 古琴聲光秀 第二步「硬體採購與打樣」計畫

## Context / 背景

`計畫─第一步.md` §7 已定出第二步的方向性選擇（LED 用 SK6812/WS2812B、建議 144 顆/米；控制器用 ESP32），但明確標註「此步需另立計畫並先做硬體採購/打樣評估」——也就是說在買齊七條 120cm 燈條之前，應先用小規模打樣驗證解析度、訊號穩定度與亮度/耗電是否符合預期，避免一次大量採購後才發現規格不合用。

本計畫只涵蓋**打樣階段**：驗證單條短燈條 + ESP32 能否正確重現第一步匯出的 Project JSON 光效。正式量產（七條 120cm 全長採購、外殼/固定方式、與音檔同步播放的完整實作）待打樣結果出來後再另立計畫。

---

## 1. 打樣目標（需驗證的問題）

1. **解析度是否足夠**：144/m 在最窄徽位間隔（二徽↔三徽，約 40mm，理論上約 5.8 顆 LED）滑動走手音時，肉眼看起來是否夠平滑；搭配 §7 提到的次像素混色（sub-pixel blending）能否進一步改善。
2. **訊號穩定度**：ESP32（3.3V 邏輯）直接驅動 SK6812/WS2812B（建議 5V 訊號）在短距離是否會出現閃爍/花色等訊號不穩現象，是否需要加電平轉換模組。
3. **亮度與耗電**：實際量測單顆 LED 在專案會用到的顏色/亮度下的電流，換算七條 120cm、144/m 滿載情境下需要多大的電源供應器（瓦數/安培數）。
4. **軟體對接**：確認 FastLED（或 WLED）能正確讀取第一步匯出的「編譯後逐影格逐燈亮度」資料並即時播放，與音檔同步的誤差是否在可接受範圍（沿用第一步 §9 的 <~50ms 對齊標準）。

---

## 2. 打樣採購清單（小量，非正式量產）

| 項目 | 建議規格 | 數量 | 用途 |
|---|---|---|---|
| 可定址 LED 燈條 | SK6812（或 WS2812B），144 LED/m，5V | 1 米（可裁切測試不同段落） | 驗證解析度與滑動平滑度 |
| 微控制器開發板 | ESP32 開發板（如 ESP32-DevKitC，常見型號即可） | 1 片 | 跑 FastLED/WLED，讀取編譯後燈效資料 |
| 電平轉換模組 | 74AHCT125（單向三態緩衝器，接線較簡單，推薦優先試）或 74HCT245（雙向匯流排收發器，`DIR` 腳固定接高、只用單向；社群另一常見選擇） | 各 1 片 | 測試「有無轉換」訊號穩定度差異，可先跳過用軟體 workaround（如把訊號腳走短線）比較；兩者都是靠 HCT 系列輸入門檻相容 3.3V，本質上是同一招，差別只在接線複雜度 |
| 5V 電源供應器（打樣用） | 5V / 3–5A 即可（單米燈條全白滿亮約需 144×60mA≈8.6A，抓保守值） | 1 個 | 供電，量測實際電流 |
| 濾波電解電容 | 1000µF、6.3V 以上耐壓 | 2–3 顆 | 接在燈條電源輸入端，防止上電衝擊 |
| 訊號串接電阻 | 300–500Ω | 2–3 顆 | 接在 ESP32 訊號腳到燈條資料線之間 |
| 杜邦線 / 麵包板 | 常見公母杜邦線一組 + 一片麵包板 | 1 組 | 打樣接線，不需焊接 |
| USB 電表 / 萬用電表 | 能量測電流、電壓即可 | 1 個 | 實測耗電，換算量產電源規格 |
| 音樂古琴錄音樣本 + 第一步軟體匯出的 JSON | 沿用第一步驗證用的錄音 | — | 打樣播放內容來源 |

> 這些是打樣規模（1 米燈條、1 片開發板），刻意不先買七條 120cm——先驗證原理與參數，量產規格（總米數、確切電源瓦數、是否分腳位供電等）待第 3 節打樣完成後再定案。

---

## 3. 打樣步驟

1. **接線**：
   - **電源**：5V 電源供應器直接接燈條電源輸入（起始端加濾波電容），不經過 ESP32；ESP32 另外用 USB 或 VIN 供電（板載穩壓器會自行降到晶片實際吃的 3.3V）；燈條電源的 GND、ESP32 的 GND 全部接在一起（共地，訊號判讀的電位基準才會一致）。
   - **訊號**：ESP32 訊號腳 →（可選）電平轉換 → 300–500Ω 電阻 → 燈條資料輸入。
   - **若使用 74AHCT125 電平轉換**，額外要接：`VCC` 接 5V（與燈條同一電源即可，不需另外一組）、`GND` 共地、輸入腳 `A` 接 ESP32 訊號腳、輸出腳 `Y` 接電阻後再到燈條資料輸入、**`OE`（輸出致能）務必接 GND**（它是低電位致能，懸空或接高會導致輸出呈高阻抗、完全沒訊號通過——這是最容易漏接的一腳）。
2. **燒錄測試程式**：先用 FastLED 內建 demo（跑馬燈/彩虹）確認硬體通訊正常，排除接線問題。
3. **接入編譯後資料**：從第一步軟體匯出的「編譯成 LED 影格」資料（§5 提到的逐影格逐燈亮度序列），寫一段簡單腳本轉成 ESP32 可讀格式（如陣列/SPIFFS 檔案），在 ESP32 上依影格率播放。
4. **同步播放測試**：ESP32 播放燈效的同時，另一台裝置（或同一台）播放對應音檔，肉眼/耳朵評估同步誤差；量測滑音段落的視覺平滑度。
5. **耗電量測**：用電表記錄不同亮度/顏色下的實際電流，反推七條 120cm、144/m 滿載時需要的電源瓦數（附加安全餘裕，建議抓理論值的 1.3–1.5 倍）。
6. **訊號穩定度比較**：分別在「有/無電平轉換」情境下觀察短距離與加長線材（如延伸到 1.2 米）時是否出現閃爍。

---

## 4. 打樣後才能定案的事項（本計畫不先假設答案）

- 七條 120cm 燈條的確切採購來源與單價（打樣後依實測規格找供應商）。
- 正式電源方案：單一大電源 + 分路供電，或每條燈條獨立小電源。
- 訊號走線方式：七條是否共用一個資料腳位（菊鏈）或各自佔用 ESP32 一個 GPIO（影響是否需要多顆 ESP32 或訊號分配器）。
- 古琴本體/骨架的固定方式與材質（將光帶嵌入或黏貼的實體結構，本計畫未涵蓋）。
- 是否需要外接電源管理板（如降壓模組）视 ESP32 供電來源（USB 或獨立電源）而定。

---

## 5. 驗證方式（打樣完成的判準）

- 单条 1 米燈條能正確播放第一步匯出的編譯後資料，滑音視覺平滑（無明顯跳格）。
- 音效與燈效同步誤差 < ~50ms（沿用第一步標準）。
- 量測到的實際電流數據已記錄，可據此換算七條全長的電源需求。
- 已確認訊號穩定度足夠（或已確認需要電平轉換模組並驗證加入後解決問題）。

> 打樣完成、上述判準都通過後，再依實測結果另立「正式量產與組裝」計畫，屆時才決定七條全長燈條的正式採購清單。
